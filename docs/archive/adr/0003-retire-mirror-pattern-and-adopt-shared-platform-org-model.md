# ADR 0003 — Retire the Mirror-Pattern Schema and Adopt the Shared `platform` Org Model

- **Status:** Accepted
- **Date:** 2026-07-14
- **Decided by:** LiNKplatform build sequence step 2 (`LiNKplatform/docs/specs/shared-foundation-spec.md` §10, §11), executed as a LiNKsites-side retrofit
- **Context source:** `LiNKplatform/docs/specs/shared-foundation-spec.md` (§1–4, §10–11), `supabase/migrations/20260331_000001_lsites_init.sql` (old mirror-pattern schema, `lsites_core`), `supabase/migrations/20260714_000001_program_ledger_core.sql` (Program Ledger, `lsites_ledger` — unaffected by this ADR), `docs/archive/adr/0002-linkskills-capability-lease-exemption.md`

## Context

LiNKtrend's original LiNKsites data architecture used a **mirror-database pattern**: Payload CMS had its own Postgres database (`public` schema, `postgres-payload` per `audit/04_data_and_storage_registry.yaml`), and a separate Supabase-hosted schema (`lsites_core`, defined in `supabase/migrations/20260331_000001_lsites_init.sql`) held a parallel copy of `sites`, `pages`, `navigation`, and related content, plus `sync_ingress`/`sync_jobs` tables explicitly documented as receiving data `from lautowork`. Tenant isolation in `lsites_core` was enforced entirely through a single session variable, `current_setting('app.site_id', true)`, not through any real tenant/organization RBAC model.

This architecture has been superseded company-wide. The decided target architecture (per `shared-foundation-spec.md` §1) is: no database inside Payload CMS itself as a second source of truth, exactly one Postgres database per environment (hosted on Supabase, in the shared `linkplatform-stage`/`linkplatform-prod` projects per spec §2), with the frontend caching content so most visits don't hit the database per request. `LiNKplatform` has now shipped the shared cross-Program foundation this depends on: `platform.organizations`, `platform.org_members`, `platform.capabilities`, `platform.capability_grants`, `platform.handoff_envelopes`, and the `platform.has_org_access(org_id, min_role)` RLS helper (`LiNKplatform/supabase/migrations/20260714_000001_platform_foundation.sql`). Spec §10 step 2 explicitly names this retrofit: "Retrofit LiNKsites onto the shared project — add `org_id` to its tenant tables, wire its capability leases to check `platform.capability_grants`, retire its old mirror-pattern migration (`lsites_init.sql`), and supersede LiNKsites ADR 0002."

Two things had to be established before deciding how to retire `lsites_core`: (1) whether it is actually wired into any running application code today, and (2) how LiNKsites' tenant scoping should change once real `platform.organizations` identity exists.

## Finding: `lsites_core` is dormant scaffolding, not live-wired into running application code

A repo-wide search for `lsites_core`, `sync_ingress`, `sync_jobs`, and `app.site_id` found five categories of hits, none of them in a request-serving code path:

1. **`apps/cms/src/payload.config.ts`** — actively **avoids** `lsites_core`. Its `DATABASE_URI` handling forces `search_path=public` for any Supabase-hosted connection string that would otherwise default to `lsites_core`, with a comment explaining that "Payload collections live in `public`." Payload's own tables (the actual content served to visitors, per `postgres-payload` in `audit/04_data_and_storage_registry.yaml`) have never lived in `lsites_core`.
2. **Standalone, manually-invoked scripts only** — `apps/cms/scripts/sync-supabase-to-cms.ts` (`pnpm supabase:sync`) and `apps/cms/scripts/seed-supabase-lsites-core.ts` (`pnpm supabase:seed`) are the only code that opens a raw `pg` connection and runs `set search_path to lsites_core, public`. Both are npm scripts a human or a future automation would run on demand; neither is invoked by the CMS server, by `apps/web-master`, by any CI workflow (`.github/` has zero matches for either script name or for `lsites_core`), or by any cron/n8n wiring found in the repo.
3. **`packages/types/src/data-contract.ts`** re-exports a generated JSON schema of `lsites_core` and a CMS field mapping. Its only consumers, per repo-wide search, are two more of the same standalone scripts (`sync-supabase-seed.ts`, `validate-contract.ts`) — not any app runtime code.
4. **`docs/archive/reference/SUPABASE_SETUP.md`** and **`supabase/schemas/lsites_core.schema.json`** — documentation and a generated schema artifact, not executable app logic.
5. **`audit/04_data_and_storage_registry.yaml`** and **`audit/05_current_architecture.md`** (Phase 0 audit, 2026-07-13) independently confirm this: "No `@supabase/supabase-js` client anywhere — accessed as plain Postgres, not via Supabase SDK/Data API in code," and list `lsites_core`'s only consumers as "`svc-payload-cms` (if co-located)" (conditional, unverified) and "sync scripts under `apps/cms/scripts/`" — i.e., the audit reached the same dormant-scaffolding conclusion independently, before this ADR.

**Conclusion: `lsites_core` is unused-in-production scaffolding.** No visitor request, no Payload admin action, and no automated pipeline reads or writes it today. It exists as a design artifact for a mirror-sync flow that documentation describes but that no live system triggers. This is materially different from `lsites_ledger` (`20260714_000001_program_ledger_core.sql`), whose own migration comment states plainly "no application code reads/writes these tables yet" for the *opposite* reason — it's a forward-looking schema for a not-yet-wired-in future store, not a legacy pattern being phased out. `lsites_ledger` is not part of the mirror pattern and this ADR does not touch it.

Because `lsites_core` is dormant rather than live-wired, **this ADR recommends straightforward retirement**, not a staged dual-run cutover: there is no live traffic to protect during a transition, so the added process overhead of a parallel-running interim state would be pure cost with no corresponding risk reduction. If a future audit or code change discovers this finding was wrong — e.g., some deployed environment's CMS *is* actually configured with `search_path=lsites_core` in practice, contradicting the code default — that changes the risk profile materially and this recommendation should be revisited before executing the retirement steps below.

## Decision 1 — Retire `lsites_core`, adopt shared `platform` schema for tenant identity

- `lsites_core` (the old mirror-pattern schema, defined in `supabase/migrations/20260331_000001_lsites_init.sql`) is retired. LiNKsites will not maintain its own local tenant/organization table. Tenant identity going forward is `platform.organizations` / `platform.org_members`, owned by `LiNKplatform` and shared across every Program (spec §4).
- Recommended retirement mechanics, per this repo's own "archive before delete" policy (`docs/archive/DOCUMENTATION_GOVERNANCE.md` §Relevance Policy: "If obsolete and superseded: archive first, then schedule deletion in a later cleanup PR"):
  1. Write a new, separately-dated **down-migration** that drops the `lsites_core` schema (mirroring the `-- migrate:down` block already present at the bottom of `20260331_000001_lsites_init.sql`, which already contains a complete, working `drop schema if exists lsites_core` sequence — it can be lifted almost verbatim into its own migration file rather than re-derived).
  2. **Archive, don't delete,** `20260331_000001_lsites_init.sql` itself — move it to `supabase/migrations/archive/` (or `docs/archive/` if the migrations directory has no existing archive convention; either is consistent with the governance doc's "archive first" instruction) with a short header comment noting it was superseded by this ADR and by the shared `platform` schema. Do not hard-delete it — it remains the historical record of the original mirror-pattern design.
  3. Retire the standalone sync/seed scripts (`sync-supabase-to-cms.ts`, `seed-supabase-lsites-core.ts`, `sync-supabase-seed.ts`, `validate-contract.ts`, `validate-supabase.ts`) and their `package.json` script entries in the same change, since they have no target schema left to read from. Archive rather than delete if any of them contain logic worth preserving as reference (e.g., the Payload-upsert helper functions in `sync-supabase-to-cms.ts` could be adapted for a future, real content-population pipeline).
  4. Update `docs/archive/reference/SUPABASE_SETUP.md`'s "LiNKsites Schema + Search Path" section, which currently instructs every connection string to set `search_path=lsites_core,public` — this is now wrong under the new architecture and should instead describe connecting to the shared `linkplatform-stage`/`linkplatform-prod` project with `search_path` scoped to whatever schema LiNKsites' own working-layer tables end up in (see Decision 2) plus `public` for Payload's own tables.
  5. Update `audit/04_data_and_storage_registry.yaml`'s `postgres-supabase-lsites_core` entry to reflect retirement rather than leaving it describing a live store.
- **Resolved (2026-07-15): confirmed no live LiNKsites data exists anywhere.** `20260331_000001_lsites_init.sql` was archived without ever being applied to `linkplatform-stage` or `linkplatform-prod` — no data migration is needed. `20260715_000001_lsites_sites_core.sql` is therefore a fresh `create schema`/`create table`, not a data-preserving `alter`, and was applied directly to `linkplatform-stage` in the same session this ADR's open item was closed.

## Decision 2 — Real `org_id` on `sites`, not a repeated `org_id` on every child table

LiNKsites' tenant scoping today is `current_setting('app.site_id', true)` — a single session variable, keyed to one *site*, with no notion of an *organization* (a tenant client account may own more than one site) and no real RBAC (any connection holding the runtime role and the right session variable has full access; there's no per-user role check).

**Decision (finalized 2026-07-15): add `org_id uuid references platform.organizations(id)` to the top-level `sites` table only**, in the schema's replacement — named `lsites_sites` (per the schema-per-Program prefixing convention in spec §3, since a bare `lsites_core` no longer makes sense once there's no "core vs. ledger" split left to distinguish). **Do not add `org_id` to child tables** (`pages`, `articles`, `media`, `navigation`, `site_settings`, and the other `site_id`-scoped tables already cascading from `sites`). Implemented in `supabase/migrations/20260715_000001_lsites_sites_core.sql`.

Rationale for this specific split:

- Every child table already has a `not null` (or nullable-but-always-populated-in-practice) foreign key to `sites.id` via `site_id`, with `on delete cascade`. Organization isolation for a child row is fully determined by which site it belongs to — a `pages` row cannot belong to a different organization than its parent `sites` row without also being an orphaned/dangling reference, which the existing FK already prevents. Duplicating `org_id` onto every child table would be redundant data that could drift from the parent (e.g., an `UPDATE` to a child row's `org_id` without a matching update to the parent) — a real integrity risk with no corresponding benefit, since the parent's `org_id` is always one join away.
- RLS for child tables should be a **join-based check against `sites`**, using `platform.has_org_access()` transitively: `exists (select 1 from lsites_sites s where s.id = pages.site_id and platform.has_org_access(s.org_id, 'client_viewer'))`. This keeps the "one membership check, defined once in `platform`" property spec §4 asks for (no Program duplicating membership logic), while still scoping every table correctly through the existing `site_id` foreign key.
- The existing `current_setting('app.site_id', true)` session-variable pattern is **not discarded outright** — it can continue to coexist as a *performance* optimization (avoiding a join on every single-site-scoped query when a request already knows which site it's operating on) or be retired entirely once the join-based policies are proven correct; either is an acceptable follow-on implementation choice. What must change is that it is no longer the *only* mechanism, and it is no longer keyed to a bare site UUID with no relationship to a real, authenticated user's organization membership — `platform.org_members`/`platform.has_org_access()` is the actual authority for "is this caller allowed to touch this tenant's data," with the session-variable pattern (if kept) narrowing further within an already-authorized scope.
- This also matches the general shape `shared-foundation-spec.md` §4 describes ("Every Program-owned table that carries tenant data includes `org_id`") without over-reading it as "literally every single table" — the spec's own worked example (`platform.capability_grants`, `platform.handoff_envelopes`) puts `org_id` on top-level tenant-scoped records, not on every downstream row that already has an unambiguous path back to one.

Executed in `supabase/migrations/20260715_000001_lsites_sites_core.sql` (the earlier `DRAFT_platform_org_retrofit.sql` is removed, superseded by this real, dated migration once the fresh-creation approach above made an ALTER-based retrofit unnecessary):
- `org_id` added to `sites`, nullable (no backfill needed — zero existing rows).
- `sites`' own RLS policy checks `platform.has_org_access(org_id, ...)` OR'd with the original `app.site_id` session-variable check (either passing is sufficient) — `sites` had no RLS at all in the archived original; it does now.
- Every child table keeps its `site_id`-scoped access path and gains an additional join-based `platform.has_org_access()` check through `sites`, generated once via a `do $$ ... foreach ... $$` loop rather than repeated by hand per table. No child table gained a redundant `org_id` column.

## Decision 3 — Supersede ADR 0002: LiNKsites' Program Ledger is the permanent home for its own governance/leases

ADR 0002 framed LiNKsites' exemption from LinkSkills capability leases as **interim**, with an explicit revisit trigger once LiNKsites built its own Program Ledger. Two things have changed since:

1. LiNKsites' Program Ledger (`lsites_ledger` schema, `packages/program-ledger`) is now built — not just as a schema, but with real Issue/Run/Gate/Event/idempotency logic, tests, and at least one real `ExecutorAdapter` connecting it to actual factory-catalog work (per `audit/14_implementation_roadmap.md`'s Phase 2/3 batches). The revisit trigger ADR 0002 named has fired.
2. `shared-foundation-spec.md` §5 and §11 resolve the underlying tension ADR 0002 was tracking, but in the **opposite direction** its "interim exemption, revisit later" framing implied. LinkSkills is not being extended to cover governance or leases for any Program — it is permanently scoped to "catalog + mandatory-eval-suite + telemetry only" (spec §7: `lskills_catalog`, `lskills_telemetry`, `lskills_eval_runs`). Governance/lease enforcement is explicitly a `LiNKplatform` + per-Program-Ledger responsibility instead (spec §5: "each Program still enforces its own leases/gates via its own Program Ledger; [the `platform.capabilities`/`platform.capability_grants`] registry is the catalog + licensing policy those Ledgers check against"). This is stated as the corrected, permanent design ("Replaces the earlier (incorrect) idea of LinkSkills owning governance"), not a temporary gap.

**Decision: LiNKsites' own Program Ledger (`lsites_ledger`) is, and remains, the permanent home for its governance and capability-lease enforcement.** No future LinkSkills integration for LiNKsites' leases should be expected or planned. The one integration point that *is* expected going forward is narrower and one-directional: LiNKsites' Program Ledger should check `platform.capability_grants` (via `org_id`) before issuing a lease for anything touching an external system, per spec §5 — this is a licensing-catalog lookup against `LiNKplatform`, not a governance hand-off to LinkSkills.

This ADR formally supersedes ADR 0002. ADR 0002's Status line has been updated to "Superseded by ADR 0003"; its body is left untouched as a historical record of the original interim decision and rationale.

## Consequences

- `lsites_core` retirement removes a dormant, unmaintained mirror-sync surface with no live consumers, at low risk, once the archive-then-drop sequence in Decision 1 is executed.
- `sites` gains a real `org_id` column tying it to `platform.organizations`; child tables do not, keeping the change minimal and avoiding data-integrity duplication.
- LiNKsites' RLS model moves from a single session-variable trust boundary to a real, auditable membership check (`platform.has_org_access()`), while retaining the option to keep `app.site_id` as a scoping convenience layered on top.
- ADR 0002 is closed out as superseded rather than left open-ended; there is no remaining ambiguity about whether LinkSkills will someday own LiNKsites' leases — it will not, by design.
- **Open item closed (2026-07-15):** confirmed no separate live Supabase project/data existed to migrate; `20260715_000001_lsites_sites_core.sql` was applied directly to `linkplatform-stage`.

## Alternatives considered

- **Staged/dual-run retirement of `lsites_core` (build shared model alongside, cut over, then drop):** rejected as the primary recommendation because the evidence gathered (see Finding above) shows no live traffic depends on `lsites_core` today — a dual-run period would add process overhead without reducing any real cutover risk. If that finding is later shown to be wrong for some deployed environment, this alternative should be reconsidered before executing the drop.
- **`org_id` on every tenant-scoped table, including all `site_id`-scoped children:** rejected — redundant with the existing `site_id → sites.org_id` path, introduces a drift/integrity risk, and is unnecessary once RLS uses a join-based check.
- **Keep `app.site_id` as the sole RLS mechanism, layering `platform.has_org_access()` only at the application layer instead of in RLS:** rejected — RLS is the actual trust boundary Postgres enforces regardless of application-layer bugs; pushing the org check up to the application layer only would reintroduce exactly the kind of "no real tenant/org RBAC" gap this ADR is meant to close.
- **Leave ADR 0002 open/interim rather than superseding it:** rejected — the revisit trigger it named has fired, and the shared-foundation spec resolves the question definitively rather than leaving it open; superseding it now removes a stale ambiguity rather than deferring a decision that has already been made.
