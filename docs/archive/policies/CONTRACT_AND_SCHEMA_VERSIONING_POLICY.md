# Contract and Schema Versioning Policy

**Status:** Approved — Phase 1 foundation (per `audit/14_implementation_roadmap.md`, Phase 1: "Version schemas and generated types. Define identifiers, events, contracts, configuration, and policy registries.")
**Authority:** LiNKsites Program Manual §12.17-24 (Promotion Service contracts), §18.68 (pinned lockfiles/dependency reviews), §20 (Issue/Run/contract immutability), §21.16 (contract envelope versioning), §57 (ADRs), §61 exit gate.
**Applies to:** every structured contract, schema, or generated-type surface LiNKsites defines or consumes internally (Site Specification, Vertical Kit, Tier Specification, Program Ledger records, Payload↔frontend content contracts, Supabase working-layer tables, cross-Program envelopes once reachable). This policy governs *how* future schemas are versioned; it does not itself define any new schema.

## Why this exists now

No machine-readable Site Specification, Vertical Kit, Tier Specification, or Program Ledger schema exists yet in this repository (see `audit/09_gap_and_risk_register.yaml`, GAP-01/11/12/14/15). Before any of those are built (Phase 2/3 of the roadmap), this repository needs one settled, written convention for how contracts and schemas are versioned, named, located, and tested — so every future work packet follows the same rules instead of each inventing its own.

## Core rules

1. **Every contract/schema has an explicit `schema_version` (or `version`) field.** No contract is ever assumed to be "the only version that will ever exist" — even a first version is version `1`, not implicit/unversioned. This mirrors the manual's own object glossary (§24 §92: "Each object has stable ID, schema version, lifecycle state...").
2. **Versioning scheme: `MAJOR.MINOR` integers, not full semver, for data contracts.** Data/event contracts (Site Specification, Vertical Kit, Program Ledger records, etc.) use a simple `MAJOR.MINOR` integer pair (e.g. `schema_version: 2.1`), not full semver (`2.1.3`) — patch-level schema changes should not exist; a schema either changes shape (bump MINOR for additive/backward-compatible, MAJOR for breaking) or it doesn't. **Code packages** (npm packages under `packages/*`, if/when they publish contract types) continue to use full semver per normal npm convention — this rule is about the *data contract version field*, not the package version.
3. **MAJOR bump = breaking change.** Removing a field, renaming a field, changing a field's type/meaning, or changing required-ness from optional→required is a MAJOR bump. Old MAJOR versions of a contract must either (a) continue to be accepted by consumers for a defined deprecation window, or (b) have an explicit, dated migration completed before the old version is rejected — per manual §84 (change control) and §44 (data migration/compatibility audit).
4. **MINOR bump = additive, backward-compatible change.** Adding an optional field, adding a new enum value that old consumers can safely ignore, or loosening a constraint is a MINOR bump. Existing consumers must continue to work unmodified against a MINOR-bumped producer.
5. **Every contract/schema lives in exactly one canonical location, referenced from everywhere else.** Do not define the same shape in two places (a TypeScript interface in one app and a duplicated JSON Schema in another) without one being explicitly generated from the other. Prefer this order of preference:
   - **Generated types from a single source of truth** (e.g., Payload's own `generate:types` for CMS collections — already used in `apps/cms`) over hand-written duplicate types.
   - **A shared package** (`packages/types` already exists and is the correct home for cross-app-shared contract types — see `packages/types`'s existing pattern of importing the Supabase JSON schema as a data contract) over copy-pasted types per app.
   - **JSON Schema or Zod schemas co-located with the producing service**, with consumers importing/generating from that single definition, when the contract crosses a service/deployment boundary (e.g., a future Program Ledger's Issue/Run schema, or a future cross-Program contract with Sales/Odoo/Stripe).
6. **Every schema file/module states its own version in its own content**, not only in a changelog or commit message — a schema read in isolation (e.g., by a future AI work packet, per manual §55 AI Context Package doctrine) must be able to state its own version without needing repository history.
7. **Breaking (MAJOR) changes require an ADR** (see `docs/archive/adr/`, following the pattern established by ADR 0001/0002) recording: what changed, why, the migration/compatibility plan for existing data or in-flight work, and the deprecation window for the old version. This follows manual §57 (ADRs for material choices) and §84 (change control after approval).
8. **Contract tests are mandatory for any schema crossing an app/service boundary.** Per manual §19-11.4 (Payload schema↔frontend mapping contract tests) and §38 (test audit doctrine), any schema consumed by more than one app/package needs an automated test asserting the contract's current shape — so a breaking change is caught by CI (once the CI gate covers it), not discovered at runtime. `apps/cms/tests/contracts/` is the existing, correct location for this pattern within `apps/cms`; extend the same pattern (a `tests/contracts/` or `tests/schema/` folder) to any future service that owns a contract.
9. **Identifiers are permanent; names are not.** Once a contract/schema/object type is assigned a stable identifier (e.g., `capability_id`, `gap_id`, `site_specification_id` when that object is eventually built), that identifier is never reused for a different meaning, even across major version bumps. Renaming the human-readable label is fine; reassigning the ID is not.
10. **No silent schema drift via `any`/`unknown` escape hatches.** A schema change that can't be expressed in the type system (e.g., "this field used to be a string, now it can also be a number") must be modeled explicitly (a union type, a discriminated variant, or a MAJOR version bump with a migration) — not silently absorbed with `any`/`unknown` and a runtime type-guard that isn't documented as part of the schema's own versioned shape.

## Where this applies today (concrete, not hypothetical)

- **Payload collections** (`apps/cms/src/collections/*`, `apps/cms/src/globals/*`): the canonical source of truth for CMS content shape. `payload generate:types` (already an existing script, `apps/cms/package.json`) is the correct generated-types mechanism — do not hand-write a parallel type for a Payload collection's shape elsewhere in the repo.
- **`packages/types`**: the canonical home for cross-app shared primitives (`SiteId`, `LocaleCode` already exist here, importing the Supabase JSON schema). Any new cross-app-shared contract type should be added here, not duplicated per-app.
- **`supabase/schemas/*.json`**: the existing JSON Schema files (`lsites_core.schema.json`, `cms-mapping.json`) are the correct pattern for the Supabase working-layer schema and should gain an explicit `schema_version` field if they don't already have one, as new tables/domains are added in Phase 2/3.
- **Future Program Ledger, Site Specification, Vertical Kit, Tier Specification schemas** (Phase 2/3, currently absent per GAP-01/11/12/14/15): must be built following this policy from their first version, not retrofitted after the fact.

## What this policy does not decide

- The exact schema shape of any not-yet-built object (Program Ledger, Site Specification, Vertical Kit, etc.) — those are separate, future work packets per the roadmap.
- Whether contract schemas eventually need a dedicated shared package/repo separate from `packages/types` for cross-Program contracts (Sales/Odoo/Stripe) — flagged in the Sections 07-12 extraction notes as an open question (manual doctrine doesn't mandate a specific repo location); revisit when Phase 5 (commercial spine) is scoped.
