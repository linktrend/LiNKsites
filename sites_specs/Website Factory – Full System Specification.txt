# 🌐 Website Factory System — Fully Updated Specification
**Version:** 2.0  
**Integrated With:** LinkTrend Centralized Text Content System  
**Purpose:** Multi-site, multi-language, themeable, CMS-driven Website Factory with centralized text authority.  
**Audience:** Senior engineers implementing the entire system end-to-end.

---

# 1. PURPOSE

The Website Factory allows:

1. Creating unlimited independent websites  
2. Each site is a separate GitHub repository  
3. Each site hosted on VPS infrastructure  
4. All content stored centrally in Payload’s `ContentItems` (from GitHub → n8n → Supabase → Payload)  
5. Website Factory CMS contains **only structural bindings**, never text  
6. Themes stored in frontend code; CMS stores theme ID  
7. AI enhances content generation workflows  
8. Automatic domain + SSL provisioning  
9. Versioned layouts & versioned design system  
10. Deterministic and scalable for 100+ sites

The Website Factory now uses the **Centralized Text Content System as the single source of truth** for all text.

# 2. HIGH-LEVEL SYSTEM ARCHITECTURE

## 2.1 Components

| Component | Description |
|----------|-------------|
| **Dashboard (Next.js)** | Creates sites, binds text, sets themes, manages pages, navigation, SEO, domains. |
| **Payload CMS** | Stores *structural* data + exposes `ContentItems` (central text). |
| **Supabase** | Auth for Dashboard + DB for Payload. |
| **Frontend Template Repo** | Cloned per site; contains layouts, themes, and rendering logic. |
| **Individual Site Repos** | Each website is isolated and deployed independently. |
| **Centralized Text System** | GitHub → n8n → Supabase → Payload.ContentItems. All text lives here. |
| **Redis** | ISR-like caching for dynamic content. |
| **NGINX/Traefik** | Domain routing + SSL provisioning. |
| **GitHub Actions** | CI/CD for each generated site. |
| **AI Layer** | Generates forms, sections, slot defaults, nav labels, SEO pages, meta tags, etc. |

# 3. CENTRALIZED TEXT FLOW

All canonical text flows like this:

```
GitHub “Truth Repo”
 → n8n Text Remix Pipeline
 → Supabase: content_items
 → Payload “ContentItems” Collection
 → Website Factory CMS (bindings only)
 → Frontend fetches ContentItems by doc_id + language
 → Rendered on sites
```

Website Factory **never stores text**, only:

- doc_id references  
- content block bindings  
- SEO block bindings  
- navigation label bindings

# 6. HOW CENTRALIZED TEXT SYSTEM IS CONSUMED

The Website Factory does **not** duplicate any text.

It fetches from:

### Payload → `ContentItems` collection

Filtered by:

- `doc_id`
- `language`
- `app_scope` (`shared`, `ltm-starter-kit`, etc.)
- `category` (`content_block`, `seo`, `legal`, etc.)

Example frontend fetch:

```ts
getContent(docId, lang) {
  return payload.find({
    collection: "contentItems",
    where: {
      doc_id: { equals: docId },
      language: { equals: lang }
    }
  })
}
```

The template configuration JSON (from centralized text repo) defines:

- navigation doc_ids  
- legal page doc_ids  
- SEO doc_ids  
- content block doc_ids  
- hero/feature/CTA mappings

Dashboard uses this to:

- Create pages

# 9. FRONTEND RENDERING FLOW (UPDATED)

1. Resolve site from domain or SITE_ID env  
2. Resolve language from URL/domain/cookie  
3. Resolve page slug  
4. Fetch structural data:
   - page  
   - layout  
   - slotBindings  
   - nav tree  
   - theme + overrides  

5. For each slot:
   - read `contentBindings` (doc_ids)
   - fetch text via `ContentItems`

6. Deep-merge theme variables  
7. Render layout version with slot content  
8. Cache in Redis  
9. Serve page

# 10. CACHING (UNCHANGED BUT NOW TEXT-AWARE)

Cache keys include:

- `siteId`
- `lang`
- `slug`
- layoutVersion
- themeVersion
- doc_ids used

Cache invalidation triggers when:

- Any referenced `ContentItems.doc_id` changes  
- Page/slot structure changes  
- Navigation doc_ids change

# 13. SITE CREATION WORKFLOW (UPDATED)

Dashboard → Create Site:

1. Select template  
2. Load template config JSON  
3. Pre-bind:
   - navigation doc_ids  
   - hero/feature slots doc_ids  
   - legal doc_ids  
   - SEO doc_ids  
4. Generate pages + slots with doc_id bindings  
5. Create site repo  
6. Set env vars  
7. Push repo + CI  
8. Deploy  
9. Provision domain  
10. SSL  
11. Cache invalidate  

No text is entered anywhere.

No redeploy needed.

# 17. FINAL SUMMARY OF CHANGES

The Website Factory system remains the same **structurally**, but:

### 🔥 **Slots store doc_ids, not raw text**  
### 🔥 **Navigation labels store doc_ids**  
### 🔥 **SEO pages store doc_ids**  
### 🔥 **Legal pages bind to centralized legal doc_ids**  
### 🔥 **Dashboard imports template config JSON and auto-wires bindings**  
### 🔥 **Frontend fetches all text from ContentItems, not from Website Factory**  

This achieves:

- Single source of truth for all text  
- Perfect consistency  
- Zero duplication  
- Full determinism  
- Easy multi-language  
- Future-proof AI workflows  

The Website Factory now cleanly integrates with your Centralized Text Content System.
